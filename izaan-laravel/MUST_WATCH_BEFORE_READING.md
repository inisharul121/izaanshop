# 🔄 WHY LARAVEL? (Migration from Express + Next.js)

The original Izaan Shop was built with **Express.js (backend API) + Next.js (frontend)**. That architecture required running **two Node.js processes** on cPanel, which caused:

- ❌ Node.js Passenger restarts crashing both frontend and backend
- ❌ 200MB memory limit exceeded on shared hosting
- ❌ Complex deployment (build Next.js locally, zip, upload, restart)
- ❌ ISR cache invalidation complexity
- ❌ Not cPanel-friendly (Node apps are second-class on shared hosting)

The **Laravel rewrite** solves all of this:

- ✅ PHP is a **first-class citizen** on cPanel shared hosting
- ✅ **No Node.js required at runtime** — PHP handles everything
- ✅ Blade templates = server-rendered HTML, no client hydration issues
- ✅ Laravel + MySQL = battle-tested, low memory footprint
- ✅ Same database — same PascalCase tables, same camelCase column names

# ⚠️ CRITICAL: No Node.js on cPanel

## Why You Don't Need Node to Run This App

Vite (Node.js tool) is used **only locally** to compile CSS and JS into `public/build/`. Once built, the compiled files are committed and uploaded to cPanel. PHP's `@vite()` directive reads `public/build/manifest.json` — **zero Node.js on the server**.

```
Correct workflow:
  [LOCAL] npm run build → generates public/build/
  [LOCAL] git add public/build/ && git commit
  [CPANEL] git pull (or upload files)
  [CPANEL] php artisan config:cache
  Done — NO NODE NEEDED!

Wrong workflow (what you were doing before):
  [CPANEL] npm run dev   ← DELETE THIS. Never run this on cPanel.
```

## Why `public/hot` File Causes Node Requirement

If a `public/hot` file exists, Laravel loads CSS/JS from `http://127.0.0.1:5173` (Vite dev server). **This file is auto-generated by `npm run dev`** and must NEVER be uploaded to cPanel.

The `.gitignore` already excludes `public/hot`. The `public/build/` folder is intentionally **NOT** in `.gitignore` — it must be committed.

# 🏗️ SYSTEM ARCHITECTURE

## Two-Layer Rendering (Simple & Server-Side)

Unlike the old Next.js app's 3-layer SSR strategy, Laravel uses a simpler approach:

```
Request arrives at cPanel (PHP/Apache)
    │
    ▼
Laravel Router (routes/web.php)
    │
    ▼
Controller (fetches data from MySQL via Eloquent)
    │
    ▼
Blade Template (renders complete HTML server-side)
    │
    ▼
Response sent to browser (full HTML, no hydration)
```

# 🗄️ DATABASE RULES (PascalCase Tables!)

## CRITICAL: The database was created by Drizzle ORM (Node.js)

All table names are **PascalCase** and all column names are **camelCase** because the original Express/Drizzle schema used JavaScript conventions. Every Eloquent model MUST declare these explicitly.

### ✅ Correct Model Pattern

```php
class Product extends Model
{
    protected $table = 'Product';        // ← PascalCase, REQUIRED
    const CREATED_AT = 'createdAt';      // ← camelCase, REQUIRED
    const UPDATED_AT = 'updatedAt';      // ← camelCase, REQUIRED

    protected $fillable = ['name', 'slug', 'price', 'categoryId', ...];
}
```

### ❌ Never Do This

```php
class Product extends Model
{
    // Missing $table → Laravel defaults to 'products' → WRONG TABLE
    // Missing CREATED_AT → Laravel uses 'created_at' → WRONG COLUMN
}
```

### Table Name Reference

| Model | Table Name | Key Columns (camelCase) |
|-------|-----------|------------------------|
| User | `User` | `createdAt`, `updatedAt`, `role`, `isApproved` |
| Product | `Product` | `categoryId`, `isFeatured`, `isDeleted`, `salePrice`, `createdAt` |
| Category | `Category` | `createdAt`, `updatedAt` |
| Order | `Order` | `userId`, `shippingMethodId`, `createdAt` |
| OrderItem | `OrderItem` | `orderId`, `productId` |
| Banner | `Banner` | `isActive`, `createdAt` |
| ShippingMethod | `ShippingMethod` | `isActive` |
| Coupon | `Coupon` | `isActive`, `usageLimit` |

### Validation: Use Actual Table Names

When using `exists:` validation rules, use the **exact PascalCase table name**:

```php
// ✅ Correct
'categoryId' => 'required|exists:Category,id',

// ❌ Wrong (table doesn't exist)
'categoryId' => 'required|exists:categories,id',
```

## How Laravel Runs on cPanel

cPanel runs PHP via **Apache mod_php** or **PHP-FPM** — no Node.js, no Passenger. Laravel is perfectly supported:

```
Browser Request → Apache → public/index.php → Laravel → PHP → MySQL → Response
```

**cPanel Details:**
- **Domain:** izaanshop.com
- **Document Root:** `/home/izaansho/public_html`
- **Laravel Root:** `/home/izaansho/izaan-laravel` (or inside public_html)
- **PHP Version:** 8.2+ (set in cPanel → PHP Version Selector)
- **Database:** izaansho_db (MySQL)

## cPanel File Structure (Recommended)

```
/home/izaansho/
├── public_html/           ← Apache serves THIS
│   ├── index.php          ← Symlink or copy from Laravel's public/
│   ├── .htaccess          ← Laravel's mod_rewrite rules
│   ├── build/             ← Compiled CSS/JS
│   └── images/            ← Static assets
│
└── izaan-laravel/         ← Laravel app (ABOVE public_html)
    ├── app/
    ├── vendor/
    ├── .env               ← Secret config (never public)
    └── ...
```

### Option A: Quick (public in public_html)

Point cPanel document root to `/home/izaansho/izaan-laravel/public`

# 🤖 FOR AI ASSISTANTS (GPT/Claude)

## MANDATORY CONTEXT — Read Before Any Change

### 1. Database is NOT Laravel-native

The database was created by **Drizzle ORM (Node.js)**. Every table uses **PascalCase names** (`Product`, `Category`, `User`) and every column uses **camelCase** (`categoryId`, `createdAt`, `isDeleted`). This is the single most common source of bugs.

**Always verify before touching a model:**
```php
protected $table = 'Product';     // Must match exactly
const CREATED_AT = 'createdAt';   // Must match exactly
const UPDATED_AT = 'updatedAt';   // Must match exactly
```

### 2. No Node.js Runtime Required

Never suggest running `npm` on the cPanel server. Assets are pre-compiled locally and committed to git. The build workflow is:
1. Developer runs `npm run build` locally
2. Commits `public/build/` to git
3. cPanel does `git pull` — no Node needed

### 3. No React, No Next.js

This is a pure Blade + Alpine.js application. There is no `useState`, no `useEffect`, no `fetch` calls from JavaScript (except Alpine.js reactivity). All data fetching is done in PHP controllers.

### 4. Alpine.js for Interactivity

Complex UI features use Alpine.js (`x-data`, `x-show`, `x-transition`, etc.). It's loaded via CDN in the main layout. Use Alpine for: sliders, mobile drawers, dropdowns, toggling visibility.

### 5. Tailwind Classes Must Be in Blade Files

Tailwind purges unused classes at build time. If you add a new Tailwind class, it must be in a `.blade.php` file that's listed in `tailwind.config.js` content paths. After adding new classes, run `npm run build`.

### Files You'll Interact With Most

**Controllers:**
- `app/Http/Controllers/ShopController.php` — homepage, filters
- `app/Http/Controllers/Admin/ProductController.php` — admin product CRUD
- `app/Http/Controllers/CartController.php` — cart logic
- `app/Http/Controllers/CheckoutController.php` — order creation

**Models:**
- `app/Models/Product.php` — most complex model
- `app/Models/Order.php` — order + relationships

**Views:**
- `resources/views/shop.blade.php` — homepage with sidebar
- `resources/views/components/filter-sidebar.blade.php` — price/category filter
- `resources/views/components/hero-slider.blade.php` — banner carousel
- `resources/views/layouts/shop.blade.php` — main layout

**Config:**
- `.env` — database, app settings
- `routes/web.php` — all URL routes

### Common AI Tasks

| Task | Approach |
|------|----------|
| Add product field | 1. Add to `$fillable` in Product model 2. Update create/edit Blade forms 3. Update controller validation |
| Add admin page | 1. Create controller in `Admin/` 2. Create view in `resources/views/admin/` 3. Add route in `web.php` under admin middleware group |
| Fix filter bug | Check `ShopController::index()` — all filters (category, keyword, minPrice, maxPrice, sort) must be handled |
| Add new Blade component | Create `resources/views/components/my-component.blade.php`, use as `<x-my-component />` |
| Debug 500 error | Check `storage/logs/laravel.log` — `tail -50 storage/logs/laravel.log` |

### Red Flags (Stop and Ask)

- Changing any `$table` or `CREATED_AT` constant → verify against live DB first
- Running `npm` on cPanel → **NEVER**
- Adding `php artisan migrate` on production → backup DB first
- Touching `bootstrap/app.php` → affects entire framework bootstrapping
- Removing `public/build/` from git → will break cPanel deployment

## Code Style Rules

### PHP / Laravel
- Controllers: PascalCase filenames, return `view()` or `response()->json()`
- Models: Always set `$table`, `CREATED_AT`, `UPDATED_AT`
- Routes: `web.php` for HTML, `api.php` for JSON responses
- Validation: `$request->validate()` in controllers, use exact DB table names in `exists:`

### Blade Templates
- Use Alpine.js (`x-data`, `x-show`) for interactivity
- Pass data from controller using `view('name', compact('products', 'categories'))`
- Use `@props(['var' => 'default'])` at top of anonymous components
- Use `{{ $var }}` for escaped output, `{!! $var !!}` only for trusted HTML

### CSS / Tailwind
- Customize colors/fonts in `tailwind.config.js` → `theme.extend`
- Custom components go in `resources/css/app.css` → `@layer components`
- Never inline styles unless absolutely necessary

# ✅ COMMON TASKS & SOLUTIONS

## Task: Add a New Product Field

1. **Add to model fillable**
```php
// app/Models/Product.php
protected $fillable = [
    'name', 'slug', 'price', 'newField', ...  // ← add here
];
```

2. **Create migration**
```bash
php artisan make:migration add_new_field_to_product_table
```

3. **Update create/edit Blade form**
```html
<input name="newField" value="{{ old('newField', $product->newField ?? '') }}">
```

4. **Update controller validation**
```php
'newField' => 'nullable|string|max:255',
```

5. **Deploy**
```bash
npm run build && git add . && git commit -m "Add newField to products"
# On cPanel: git pull && php artisan migrate && php artisan view:cache
```

## Task: Debug 500 Error

```bash
# 1. Check Laravel logs
tail -100 storage/logs/laravel.log

# 2. Enable debug mode temporarily (.env)
APP_DEBUG=true

# 3. Check PHP syntax
php -l app/Http/Controllers/MyController.php

# 4. Clear all caches
php artisan cache:clear
php artisan config:clear
php artisan view:clear
```

## Task: CSS/Styling Not Updating

```bash
# Rebuild assets
npm run build

# If still not working — hard refresh browser (Ctrl+Shift+R)
# Check public/build/manifest.json timestamp
```

## Task: Admin Login Not Working

Common reasons:
1. User `role` column is not `'admin'` in DB
2. `AdminMiddleware` not registered in `bootstrap/app.php`
3. `auth()->user()->role` → check actual column value in DB

```sql
-- Check admin user in DB
SELECT id, email, role FROM User WHERE email = 'admin@example.com';
UPDATE User SET role = 'admin' WHERE email = 'admin@example.com';
```

## Task: Filter Sidebar Not Showing

Ensure ShopController passes `categories` and `storeMaxPrice` to the view, and the `filter-sidebar` component starts with:
```php
@props(['categories', 'storeMaxPrice' => 5000])
```
